# Puzzle Graph 演算法設計

將解謎遊戲建模為**有向無環圖 (DAG)**，可解性在生成階段數學性地保證。

---

## 核心概念

### 實體類型

| 實體 | 說明 |
|---|---|
| **Room** | 空間容器，包含地板物品和鎖 |
| **Item** | 物品：key（消耗型鑰匙）、clue（線索）、tool（可重複使用工具） |
| **Lock** | 機關：container（容器鎖，保護物品）或 spatial（空間鎖，連接房間） |

### 鎖機制

| 機制 | 說明 |
|---|---|
| physical | 需要特定鑰匙插入 |
| password | 需要輸入密碼（配合 clue 物品） |
| hidden | 需要工具突破環境障礙（如手電筒照亮暗角落） |
| combination | 需要多個物品同時滿足 |
| minigame | 收齊零件後觸發互動式小遊戲，通過才解鎖 |

### Pickupable 屬性

| pickupable | UI 位置 | 行為 |
|---|---|---|
| true | 可見物品區 → 拾取 → 背包 | 可選取、可使用 |
| false | 機關與通道區 | 場景固定裝置，作為「使用」的目標 |

背包中的物品不區分 key/lock — 玩家可選取任何背包物品並對目標使用，系統自動判斷角色。

### 模板系統

- **鑰匙模板**：名稱、類型、是否可重複使用、是否可拾取、狀態標籤
- **鎖模板**：名稱、機制、所需鑰匙列表、分類標籤、是否可拾取、狀態標籤、小遊戲類型

每個鎖模板指定它需要哪些鑰匙模板。模板使用 **shuffle 牌堆**機制：所有模板洗牌後逐一抽取，抽完重洗，自然保證多樣性。

## 三階段生成演算法

### Phase A：房間骨架

```
建立 maxRooms 個房間（從主題池隨機取名）

對每對相鄰房間 R[i] → R[i+1]：
  從牌堆抽一個空間鎖模板
  建立空間鎖（門）
  建立對應鑰匙
  鑰匙放在 R[0..i] 範圍內（依 keySpreadRate 分散）
  記錄 criticalRoomIndex = i + 1
  加入地板物品佇列

建立出口鎖（最後一個房間）
建立出口鑰匙，加入佇列
```

**criticalRoomIndex**：每個物品記錄它必須放在哪些房間範圍內。門鑰匙的 criticalRoomIndex = 門所在房間索引 + 1，確保鑰匙永遠在門的來源側。

### Phase B：謎題內容填充

```
將地板物品佇列複製為 BFS 佇列
先將物品從房間可見清單移除（Phase B 決定最終歸宿）

當佇列非空 且 深度預算未用完：
  取出佇列首項
  若物品已在容器內 → 跳過（避免重複包裝）

  ── 模板選擇（統一流程）──
  找出物品配對的狀態鎖候選（透過狀態標籤匹配）
  過濾牌堆中所有合法的通用鎖模板（見「合法性驗證」）
  分類為：狀態鎖候選 / 組合鎖（多鑰匙）/ 單鑰匙鎖

  若有狀態鎖候選：
    按 stateLockRate 擲骰 → 選狀態鎖 或 通用鎖
  否則：
    按 compositeRate 擲骰 → 選組合鎖 或 單鑰匙鎖
  從選中的類別隨機抽一個

  若抽到模板：
    建立容器鎖（放在物品所在的房間）
    將物品放入容器
    建立子鑰匙（依 crossRoomRate 可跨房間放置），加入佇列
  否則：
    物品留在房間地板
```

**合法性驗證**（對模板的每個所需工具）：

```
若為消耗型鑰匙 → 合法
若工具就是被包裹物品本身 → 不合法（直接循環）
若工具已達復用上限 → 不合法
若工具被鎖在容器內 → 不合法（取不出來）
若工具不可拾取且不在同房間 → 不合法
若工具不在 criticalRoomIndex 範圍內 → 不合法
```

**復用路徑**：選模板前，按 reuseRate 擲骰決定是否優先復用已有工具。若命中，從場上可用工具隨機選一個，找到相容的鎖模板直接使用。

### 鑰匙建立與佇列管理

```
對鎖模板的每個所需鑰匙：
  若為可重複工具 → 從快取取得（或首次建立）
  若為消耗型 → 新建（自動避免重名）
  若不可拾取 → 強制放在鎖的同房間

  鑰匙房間 = 取 criticalRoomIndex 和 容器鎖房間索引 的較小值
  依 crossRoomRate 在合法範圍內選房間

  若工具已存在且被更嚴格的容器引用：
    收緊佇列中的 criticalRoomIndex
    必要時移動工具到合法房間
    不重新入列
```

### Phase C：整合搬移

```
計算每個實體的緊急度（被需要的鎖中最早的房間索引）
建立每個房間到達所需的門鎖鏈（用於循環檢測）

逐房間處理：
  收集地板實體（物品 + 容器鎖），按緊急度降序排列
  取得同房間的可用容器（排除狀態鎖）

  對每個候選物：
    按 consolidationRate 擲骰 → 跳過或繼續
    跳過空間鎖（門不可搬移）

    嘗試塞入每個容器：
      若 容器緊急度 > 候選緊急度 → 跳過（太晚才需要的容器不能裝太早需要的物品）
      若 候選是容器的直接或間接前置 → 跳過（循環）
        （遞迴追蹤：容器鏈 + 父容器鏈 + 門鎖鏈）
      若 容量不足 → 跳過
      若 不可攜帶物件要進可攜帶容器 → 跳過
      若 嵌套深度超過 maxNestingDepth → 跳過
      通過 → 搬入容器，從地板移除
```

## 狀態鎖

可拾取的容器鎖，代表物品的狀態轉換或合成。與一般容器鎖共用深度預算。

### 配對邏輯

```
透過狀態標籤配對：
  鑰匙模板帶狀態標籤（如手電筒: ['light-tool']）
  鎖模板帶狀態標籤（如沒電的手電筒: ['light-tool']）
  Phase B 處理物品時，若標籤有交集 → 該狀態鎖模板加入候選
```

### 範例

| 工具（產品） | 狀態鎖 | 所需鑰匙 | 場景 |
|---|---|---|---|
| 手電筒 | 沒電的手電筒 | 電池 ×2 | 裝電池 |
| 撬棍 | 斷裂的撬棍 | 金屬桿 | 修復 |
| 門禁磁卡 | 空白磁卡 | IC 晶片 | 寫入權限 |
| 破壞剪 | 鈍掉的破壞剪 | 磨刀石 | 磨利 |
| 濕布 | 乾布 | 水盆（固定） | 浸水 |

### 玩家體驗

1. 撿到狀態鎖（如沒電的手電筒）→ 放進背包
2. 找到所需鑰匙（如電池）→ 背包裡選電池 → 對手電筒使用
3. 裝滿後自動解鎖 → 有電的手電筒進背包，沒電的消失
4. 或：帶乾布到有水盆（固定裝置）的房間 → 對水盆使用 → 獲得濕布

## 小遊戲系統

小遊戲機制的容器鎖收齊零件後**不自動解鎖**，改為觸發互動式小遊戲。

```
玩家收齊所有零件（背包選取 → 對鎖使用）
鎖顯示「挑戰」按鈕
點擊 → 開啟小遊戲介面
完成 → 鎖自動解開
```

每個小遊戲鎖帶有設定（類型、隨機種子、參數），在生成階段程序化產生。

| 類型 | 說明 |
|---|---|
| pipe_puzzle | NxN 網格管道旋轉拼圖，連通入口到出口 |
| wiring | 左右端子配色接線 |

## 可調參數

| 參數 | 範圍 | 說明 |
|---|---|---|
| targetDepth | 1-20 | 全域鎖預算（含容器鎖、狀態鎖、小遊戲鎖） |
| maxRooms | 1-10 | 房間數量（固定值，非上限） |
| compositeRate | 0-1 | 選擇多鑰匙組合鎖的機率 |
| depthStaggerVariance | 0-2 | 組合鎖次要鑰匙的深度隨機偏差 |
| keySpreadRate | 0-1 | Phase A 門鑰匙分散程度（0=緊鄰門，1=任意合法房間） |
| crossRoomRate | 0-1 | Phase B 容器鎖和鑰匙跨房間機率 |
| reuseRate | 0-1 | 已有可重複工具時走復用路徑的機率 |
| maxReusesPerTool | 1-∞ | 每個工具最多被幾把鎖共用 |
| maxLocks | 1-∞ | 容器鎖總量硬上限（與 targetDepth 互補，先到先停） |
| maxNestingDepth | 0-5 | 容器最大嵌套深度 |
| consolidationRate | 0-1 | Phase C 搬入容器的機率 |
| stateLockRate | 0-1 | 有配對時選狀態鎖的機率 |

## 可解性保證

### 結構保證（生成階段）

1. **反向依賴生成**：鑰匙總是放在鎖之前可達的房間
2. **循環防護**：Phase B 合法性驗證 + Phase C 全鏈循環檢測（含容器鏈、父容器鏈、門鎖鏈）
3. **房間線性拓撲**：R0→R1→R2→... 門鎖鏈，鑰匙只能放在門的來源側或更早

### 驗證（Solver）

```
拾取所有可到達房間的可拾取物品和可拾取鎖
嘗試用背包物品開啟所有可到達的鎖（含背包中的鎖）
不可拾取物品 → 通過房間在場判定
重複直到無法繼續
若最終能打開出口 → 可解
```

自動化測試以多種配置各跑數千次，確保 100% 可解率。

## 遊戲引擎

### 雙向房間通行

空間鎖解鎖後，玩家可從門的**任一端**穿過。UI 區分：
- **本房間的門**（amber 底色 + emerald「進入」按鈕）
- **從其他房間連過來的通道**（sky 藍底色 + sky「前往」按鈕）

### MUD 指令系統

| 指令 | 縮寫 | 功能 |
|---|---|---|
| look | l | 觀察四周（含回程門） |
| inventory | i | 查看背包 |
| examine [目標] | x | 檢查物品或鎖的狀態 |
| take [物品] | t | 拾取地板物品 |
| use [物品] on [鎖] | - | 使用物品解鎖 |
| enter [密碼] on [鎖] | - | 輸入密碼 |
| go [門/房間名] | - | 穿過門到另一房間（支援雙向） |

### 物品自動回收

解鎖後掃描背包，不再被任何未解鎖需要的消耗型物品自動移除。

## 圖譜視覺化

使用 Kahn 拓撲排序計算 DAG 佈局。

### DAG 邊

| 邊類型 | 方向 | 語義 | 視覺 |
|---|---|---|---|
| requires | item → lock | 物品被鎖需要 | 實線灰色箭頭 |
| contains | lock → item | 開鎖後取得物品 | 虛線青色箭頭 |
| gate | spatial lock → 目標房間所有實體 | 過門後才能存取 | 虛線紫色箭頭（指向房間框） |

gate 邊參與排序但不畫節點間連線。可解的 puzzle 保證 gate 邊不造成循環。

### 佈局

```
Y = Kahn 拓撲 rank（深度向下），絕對值不做後處理
X = 同 rank 水平展開，按 房間順序 → 容器歸屬 排序
房間分離：計算每房間 X 範圍，依序推開確保 bounding box 不交疊
所有連線：節點底部中央 → 目標頂部中央，反向邊從底部往下繞弧
```

### 群組

| 群組 | 視覺 | 內容 |
|---|---|---|
| 房間 | 藍色邊框 | 同房間的所有節點 |
| 容器 | 紅色虛線邊框 | 容器鎖 + 遞迴 contents |

### 節點顏色

| 顏色 | 意義 |
|---|---|
| Emerald（綠） | 可拾取物品 |
| Orange（橙） | 小遊戲鎖 / 不可拾取物品 |
| Rose（玫瑰） | 容器鎖 |
| Purple（紫） | 空間鎖（門） |
| Amber（琥珀） | 出口鎖 |

圖譜始終顯示**原始 puzzle**（不受遊玩過程中的狀態修改影響）。

## 已知限制

### 房間內容分配不均

Phase B 以 FIFO 處理地板物品，大部分源自起始房間。深度預算在前面房間消耗殆盡，後面房間幾乎空的。

criticalRoomIndex 是靜態近似值，不追蹤動態依賴鏈。曾嘗試隨機房間抽選但失敗（亂數流改變 → 靜態約束無法保證可解性）。

**需要重新設計**：可能方向包括 Phase B 後處理重分配、每房間預算保留、或動態依賴追蹤。
