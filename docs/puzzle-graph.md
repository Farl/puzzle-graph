# Puzzle Graph 演算法設計

Puzzle Graph 是一種程序化生成解謎遊戲的理論框架。核心概念是將解謎遊戲建模為一個**有向無環圖 (DAG)**，其中每個謎題的可解性在生成階段就被數學性地保證。

---

## 核心概念

### 實體類型

| 實體 | 說明 |
|---|---|
| **Room** | 空間容器，包含地板物品和鎖 |
| **Item** | 物品。`key`（消耗型鑰匙）、`clue`（線索）、`tool`（可重複使用工具） |
| **Lock** | 機關。`container`（容器鎖，保護物品）或 `spatial`（空間鎖，連接房間） |

### 鎖機制

| 機制 | 說明 |
|---|---|
| `physical` | 需要特定鑰匙插入 |
| `password` | 需要輸入密碼（配合 clue 物品） |
| `hidden` | 需要工具突破環境障礙（如手電筒照亮暗角落） |
| `combination` | 需要多個物品同時滿足 |
| `minigame` | 收齊零件後觸發互動式小遊戲，通過才解鎖 |

### Pickupable 屬性

Graph 中的實體（Item 和 Lock）都有 `pickupable` 屬性，決定 UI 呈現方式：

| pickupable | UI 位置 | 行為 |
|---|---|---|
| `true` | 可見物品區 → 拾取 → 背包 | 可選取、可使用 |
| `false` | 機關與通道區 | 場景固定裝置，作為「使用」的目標 |

背包中的物品不區分 key/lock — 玩家可選取任何背包物品並對目標使用。系統自動判斷 key/lock 角色。

### 模板系統

鎖和鑰匙從預定義的模板目錄中選取：

- **鑰匙模板**：定義名稱、類型、是否可重複使用、是否可拾取、狀態標籤（用於狀態鎖配對）
- **鎖模板**：定義名稱、機制、所需鑰匙列表、分類標籤、是否可拾取、狀態標籤、小遊戲類型

每個鎖模板指定它需要哪些鑰匙模板，並帶有分類標籤用於多樣性控制。

## 兩階段生成演算法

生成分為 **Phase A（房間骨架）** 和 **Phase B（謎題內容填充）**。

### Phase A：房間骨架

建立所有房間、門鎖、出口鎖。房間數量固定為 maxRooms。

1. **建立房間**：從主題池隨機取名，產生 maxRooms 個房間
2. **建立門鎖**：相鄰房間之間各建一道空間鎖。每道門選取空間鎖模板，建立對應鑰匙。鑰匙只能放在門之前可達的房間，依 keySpreadRate 控制分散程度
3. **建立出口鎖**：放在最後一個房間，出口鑰匙放入任意合法房間

所有鑰匙加入地板物品佇列，供 Phase B 處理。

**關鍵約束 — criticalRoomIndex**：每個地板物品記錄一個 criticalRoomIndex，代表「這個物品必須放在第 0 到第 N-1 個房間的範圍內」。門鑰匙的 criticalRoomIndex 等於門所在房間的索引加一，確保鑰匙永遠在門的來源側。

### Phase B：謎題內容填充

以 BFS 處理 Phase A 的地板物品，逐一嘗試用容器鎖包裹，產生新的鑰匙加入佇列，直到深度預算用完。

**處理流程**：從佇列取出物品後，先檢查是否已在容器內（避免重複包裝）。接著判斷深度預算是否用完、是否有狀態鎖候選。若還能包裝，最多嘗試 5 次選取合法模板。狀態鎖優先（不消耗深度預算），之後才用通用模板。

**合法性驗證**：逐一檢查鎖模板所需的可重複工具。消耗型鑰匙總是合法。工具若是被包裹物品本身（直接循環）、已達復用上限、被鎖在容器內（間接循環）、不可拾取且不在同房間、或不在可達房間範圍內，則不合法。

**容器放置**：驗證通過後，依 crossRoomRate 從合法房間中隨機選擇容器鎖的位置。建立容器鎖，將物品放入，建立子鑰匙並加入佇列。

**無法包裹時**：物品直接留在房間地板。已在容器內的物品不會被覆寫位置。

### 鑰匙建立與佇列管理

每把鑰匙依模板決定類型：可重複工具從快取取得或首次建立，消耗型鑰匙每次新建（自動避免重名），不可拾取鑰匙強制放在鎖的同房間。

鑰匙的放置房間受兩個約束：不超過 criticalRoomIndex 範圍，且不超過容器鎖所在房間的索引。這確保玩家到達容器時一定能取得鑰匙。

若可重複工具已被放置且被更嚴格的容器引用，會收緊佇列中該工具的 criticalRoomIndex，必要時移動到合法房間，但不重新入列。

### Phase C：整合搬移

Phase B 結束後，將地板上的物品和小容器搬入現有的大容器中，產生多物品容器和嵌套容器。不建立新鎖或鑰匙。

先計算每個實體的**緊急度**（被需要的鎖中最早出現的房間索引），以及每個房間到達所需的門鎖鏈（用於循環檢測）。

逐房間處理地板上的實體，按緊急度降序排列（最不緊急的先嘗試搬移）。對每個候選物，依 consolidationRate 機率決定是否嘗試搬入容器。空間鎖（門）不可搬移。

搬入容器前的檢查：

| 檢查 | 條件 |
|---|---|
| 緊急度 | 容器至少要跟候選一樣緊急 |
| 循環 | 候選不能是容器的直接或間接前置（遞迴追蹤容器鏈、父容器鏈、門鎖鏈） |
| 容量 | 已用體積加上候選體積不超過容器容量 |
| 可攜帶性 | 不可攜帶的物件不能進可攜帶的容器 |
| 嵌套深度 | 不超過 maxNestingDepth |

狀態鎖（有 stateTags 的容器鎖）不作為搬入目標，因為其內容物代表特定的合成/轉換語意。

## 循環依賴防護

可重複使用的工具（tool）可以被多個鎖共用，這帶來循環依賴風險。防護分佈在 Phase B 和 Phase C 兩個階段。

### Phase B 防護

Phase B 最多嘗試 5 次不同模板，避免單次驗證失敗就放棄包裹。

**包裝前驗證**：

| 防護 | 說明 |
|---|---|
| 直接循環 | 工具就是被包裹物品本身 → 跳過 |
| 復用上限 | 工具已達每工具最大共用次數 → 跳過 |
| 容器內鎖定 | 工具被鎖在其他容器內，玩家取不出來 → 跳過 |
| 空間合法性 | 工具不在玩家可達的房間範圍內 → 跳過 |
| 重複包裝 | 已在容器內的物品不再包裝（BFS 佇列可能重複處理同一物品） |

**鑰匙入列時的防護**：

| 防護 | 說明 |
|---|---|
| 佇列收緊 | 工具被更嚴格的容器引用時，收緊佇列中該工具的房間約束，必要時移動到合法房間 |
| 鑰匙房間限制 | 鑰匙不能放在比容器鎖更遠的房間（玩家到容器時一定能到鑰匙） |

### Phase C 防護

| 防護 | 說明 |
|---|---|
| 全鏈循環檢測 | 遞迴展開容器鏈、容器的父容器鏈、空間鎖門鏈，確保候選物不是目標容器的直接或間接前置條件 |
| 狀態鎖排除 | 合成/轉換鎖不作為搬入目標，內容物語意固定 |
| 可攜帶性相容 | 不可攜帶的物件不能塞進可攜帶的容器 |

## 鎖模板選擇演算法

### 復用路徑

當 reuseRate 命中時，優先嘗試復用已有的可重複工具。從場上可用的工具（未達復用上限、未被鎖住）中隨機選一個，找出需要該工具的鎖模板。若有相容模板則使用，工具不需重新建立。

### 正常路徑

從可用鎖模板池中篩選：先匹配分類（容器或空間），再依 compositeRate 決定單鑰匙或多鑰匙，最後加權抽選。

### Tag 多樣性模式

| 模式 | 行為 |
|---|---|
| `balanced` | 使用次數少的 tag 權重更高（自動平衡各類謎題） |
| `weighted` | 按 `tagWeights` 設定的權重抽選 |
| `no-repeat` | 降低與上一次相同 tag 的權重（避免連續重複） |

## 狀態鎖（State Lock）

狀態鎖是**可拾取的容器鎖**，代表物品的狀態轉換或合成。整合在 Phase B 中，不消耗深度預算。

### 配對邏輯

透過狀態標籤配對：鑰匙模板和鎖模板各自帶有狀態標籤（如手電筒帶「光源工具」，沒電的手電筒也帶「光源工具」）。Phase B 處理一個工具時，若 stateLockRate 命中且雙方標籤有交集，就用狀態鎖包裹該工具。

### 範例

| 工具（產品） | 狀態鎖 | 所需鑰匙 | 場景 |
|---|---|---|---|
| 手電筒 `['light-tool']` | 沒電的手電筒 `['light-tool']` | 電池 ×2 | 裝電池 |
| 撬棍 `['lever-tool']` | 斷裂的撬棍 `['lever-tool']` | 金屬桿 | 修復 |
| 門禁磁卡 `['electronic-key']` | 空白磁卡 `['electronic-key']` | IC 晶片 | 寫入權限 |
| 破壞剪 `['cutting-tool']` | 鈍掉的破壞剪 `['cutting-tool']` | 磨刀石 | 磨利 |
| 濕布 `['water-station']` | 乾布 `['water-station']` | 水盆（固定） | 浸水 |

### 玩家體驗

1. 撿到狀態鎖（如沒電的手電筒）→ 放進背包
2. 找到所需鑰匙（如電池）→ 背包裡選電池 → 對手電筒使用
3. 裝滿後自動解鎖 → 有電的手電筒進背包，沒電的消失
4. 或：帶乾布到有水盆（固定裝置）的房間 → 對水盆使用 → 獲得濕布

## 小遊戲系統

小遊戲機制的容器鎖收齊零件後**不自動解鎖**，改為觸發互動式小遊戲。

### 流程

1. 玩家收齊所有需要的零件（從背包選取並對鎖使用）
2. 鎖顯示「挑戰」按鈕
3. 點擊後開啟小遊戲介面，根據設定渲染對應的小遊戲
4. 玩家完成小遊戲後鎖自動解開

### 小遊戲設定

每個小遊戲鎖帶有設定資料，包含小遊戲類型、隨機種子（確保可重現）、以及遊戲參數（如網格大小、配對數量）。設定由 MinigameRegistry 在生成階段程序化產生。

### 已實作小遊戲

| 類型 | 說明 |
|---|---|
| `pipe_puzzle` | NxN 網格管道旋轉拼圖，連通入口到出口 |
| `wiring` | 左右端子配色接線 |

## 可調參數

| 參數 | 範圍 | 說明 |
|---|---|---|
| `targetDepth` | 1-20 | 全域鎖預算（含容器鎖、狀態鎖、小遊戲鎖） |
| `maxRooms` | 1-10 | 房間數量（固定值，非上限） |
| `compositeRate` | 0-1 | 選擇多鑰匙組合鎖的機率 |
| `depthStaggerVariance` | 0-2 | 組合鎖次要鑰匙的深度隨機偏差 |
| `keySpreadRate` | 0-1 | Phase A 門鑰匙分散程度（0=緊鄰門，1=任意合法房間） |
| `crossRoomRate` | 0-1 | Phase B 容器鎖和鑰匙跨房間機率（0=同房間，1=任意合法房間） |
| `reuseRate` | 0-1 | 已有可重複工具時走復用路徑的機率 |
| `maxReusesPerTool` | 1-∞ | 每個工具最多被幾把鎖共用 |
| `maxLocks` | 1-∞ | 容器鎖總量硬上限（與 targetDepth 互補，先到先停） |
| `tagDiversityMode` | string | 鎖模板 tag 多樣性策略 |
| `tagWeights` | Record | tag 加權（僅 weighted 模式使用） |
| `stateLockRate` | 0-1 | 工具被狀態鎖包裹的機率（不消耗 targetDepth） |

## 可解性保證

### 結構保證（生成階段）

1. **反向依賴生成**：鑰匙總是放在鎖之前可達的房間（由房間索引約束）
2. **循環防護**：四層機制確保無循環依賴（見上方）
3. **房間線性拓撲**：Phase A 建立 R0→R1→R2→... 的線性門鎖鏈，鑰匙只能放在門的來源側或更早的房間

### 驗證（測試階段）

Solver 模擬玩家行為驗證可解性：
1. 拾取所有可到達房間的 pickupable 物品和 pickupable 鎖
2. 嘗試用背包物品開啟所有可到達的鎖（含背包中的 pickupable 鎖）
3. 不可拾取的物品（stationary key）通過房間在場判定
4. 重複直到無法繼續
5. 若最終能打開出口 → 可解

自動化測試以多種配置各跑 50-100 次，確保 100% 可解率。

## 遊戲引擎

### 雙向房間通行

空間鎖解鎖後，玩家可從門的**任一端**穿過。引擎判斷玩家在門的哪一端，自動計算目的地。UI 區分：
- **本房間的門**（amber 底色 + emerald「進入」按鈕）
- **從其他房間連過來的通道**（sky 藍底色 + sky「前往」按鈕）

### MUD 指令系統

| 指令 | 縮寫 | 功能 |
|---|---|---|
| `look` | `l` | 觀察四周（含回程門） |
| `inventory` | `i` | 查看背包 |
| `examine [目標]` | `x` | 檢查物品或鎖的狀態 |
| `take [物品]` | `t` | 拾取地板物品 |
| `use [物品] on [鎖]` | - | 使用物品解鎖 |
| `enter [密碼] on [鎖]` | - | 輸入密碼 |
| `go [門/房間名]` | - | 穿過門到另一房間（支援雙向） |

### 物品自動回收

解鎖後掃描背包，不再被任何未解鎖需要的消耗型物品自動移除。

## 圖譜視覺化

使用 Kahn 拓撲排序演算法計算 DAG 佈局，以鄰接表實作（O(V+E)）。

### DAG 邊的定義

| 邊類型 | 方向 | 語義 | 視覺 |
|---|---|---|---|
| `requires` | item → lock | 物品被鎖需要 | 實線灰色箭頭 |
| `contains` | lock → item | 開鎖後取得物品 | 虛線青色箭頭 |
| gate | spatial lock → 目標房間所有實體 | 過門後才能存取 | 虛線紫色箭頭（指向房間框） |

**gate 邊**參與 Kahn 排序但不顯示為節點間連線。可解的 puzzle 保證 gate 邊不會造成循環（若有循環 = 生成器 bug）。空間鎖到目標房間 group 的連線獨立繪製。

### 佈局規則

- **Y 軸**：Kahn 拓撲 rank（深度向下），絕對值，不做後處理
- **X 軸**：同 rank 水平展開，按房間順序 → 容器歸屬排序
- **房間分離**：計算每房間 X 範圍後，依序推開確保 bounding box 不交疊
- **所有連線**：從節點底部中央出發 → 目標節點頂部中央。反向邊從底部往下繞弧

### 群組

| 群組 | 視覺 | 內容 |
|---|---|---|
| 房間 | 藍色邊框 | 同 roomId 的所有節點 |
| 容器 | 紅色虛線邊框 | 容器鎖 + 遞迴 contents |

### 節點顏色

| 顏色 | 意義 |
|---|---|
| Emerald（綠） | 可拾取物品 |
| Orange（橙） | 小遊戲鎖 / 不可拾取物品（固定裝置） |
| Rose（玫瑰） | 容器鎖 |
| Purple（紫） | 空間鎖（門） |
| Amber（琥珀） | 出口鎖 |

圖譜始終顯示**原始 puzzle**（不受遊玩過程中的狀態修改影響）。

## 已知限制

### 房間內容分配不均

Phase B 以先進先出順序處理 Phase A 產出的地板物品，而大部分地板物品源自起始房間。每次包裝消耗一個深度預算，等到處理後續房間的物品時預算通常已耗盡。結果：起始房間堆滿容器鎖，後面房間幾乎空的。

房間索引約束是靜態近似值（基於房間線性順序），不追蹤動態依賴鏈。曾嘗試以隨機房間抽選取代先進先出，但失敗 — 改變亂數流後產出完全不同的 puzzle，靜態房間約束無法保證可解性（門鑰匙被搬到門後面、起點沒有任何可用資源等）。

**需要重新設計**：可能方向包括 Phase B 後處理重分配、每房間預算保留、或將靜態房間約束改為動態依賴追蹤。
