# Graph Layout 繪圖需求

## 目標

將 PuzzleDefinition 的 DAG 結構視覺化，讓設計者能一眼看出謎題的相依性和結構。

---

## 繪圖條件

### 1. 深度方向

- Y 軸代表相依性的深度，從上往下增長
- 起點（地板物品）在最上方，終點在最下方
- Y 值由拓撲排序（Kahn）決定，不做任何後處理修改

### 2. 連線方向

- 所有邊一律從節點**底部中央**出發，連到目標節點**頂部中央**
- 絕不從節點的左右側連線
- 一率加上箭頭
- 正向邊（target.y > source.y）：S 型 bezier 曲線
- 反向邊（target.y ≤ source.y）：改成對準目標頂部的箭頭根部，以方便看到

### 3. 同層排列

- 同樣深度的節點之間不重疊

### 4. 房間群組

- 每個房間用 bounding box 標示（藍色邊框），包含的鎖和鑰匙都會在框內
- 房間 bounding box 內絕對不可以有其他房間的物間，
發生的話要用x軸推開

### 5. 容器群組

- 容器鎖 + 其 contents 用紅框標示
- 框內不應包含非成員節點，發生的話要用x軸推開
- 框內會包含包含在內的容器鎖以及其容器內容

### 6. 不做的事

- 不做任何特例（逃生大門、空間鎖等都遵守相同規則）

---

## 已解決：空間鎖循環問題

**原問題**：空間鎖 → 目標房間的 gate 邊擔心造成 Kahn 循環。

**解法**：可解的 puzzle 保證不會有循環。Gate 邊直接加入 DAG，Kahn 正確計算所有深度。若出現循環 = 生成器 bug。

## 邊的定義

| 邊 | 方向 | 意義 |
|---|---|---|
| requires | key → lock | 物品被鎖需要 |
| contains | lock → content | 開鎖後取得物品 |
| gate | spatial lock → 目標房間所有實體 | 過門後才能存取（參與 rank 計算，不畫節點間連線） |

## 未解決：容器群組入侵

容器 bounding box 內可能包含非成員節點。目前的佈局用房間分欄 + 容器排序盡量避免，但無法完全消除。

---

## 節點顏色

| 顏色 | 意義 |
|---|---|
| Emerald（綠） | 可拾取物品 |
| Orange（橙） | 小遊戲鎖 / 不可拾取物品 |
| Rose（玫瑰） | 容器鎖 |
| Purple（紫） | 空間鎖 |
| Amber（琥珀） | 出口鎖 |
